# Plan: Fix Subscription Amount Decimal Truncation Bug

## Diagnosis

**Problem**: When editing a client's monthly subscription amount, entering `$60000` saves as `$6000` (one zero dropped).

**Root Cause**: The `Input` component at `src/components/modules/client-profile.tsx:786` uses `type="number"` with `parseFloat()` in the `onChange` handler.

In Argentine Spanish locale:
1. User types `60000` in the number input
2. Browser formats the display as `60.000` (dot as thousands separator)
3. When `onChange` fires, `e.target.value` returns the formatted string `60.000`
4. `parseFloat('60.000')` in JavaScript parses `.` as the **decimal separator**, returning `60` (not `60000`)
5. This `60` gets saved instead of the intended `60000`

The `$6000` symptom suggests a variant where the browser reformatting causes partial truncation.

## Solution

Change the amount input from `type="number"` to `type="text"` with proper locale-aware parsing.

### Step 1: Modify `client-profile.tsx`

**Location**: `src/components/modules/client-profile.tsx`, lines 785-791

**Change the Input component:**
```tsx
// Before:
<Input
  type="number"
  value={formData.amount ?? ''}
  onChange={(e) => updateFormData('amount', e.target.value === '' ? null : (parseFloat(e.target.value) || 0))}
  placeholder="0"
/>

// After:
<Input
  type="text"
  inputMode="numeric"
  pattern="[0-9.]*"
  value={formData.amount ?? ''}
  onChange={(e) => {
    const raw = e.target.value.replace(/[^0-9]/g, '') // strip non-digits
    updateFormData('amount', raw === '' ? null : parseInt(raw, 10) || 0)
  }}
  placeholder="0"
/>
```

This ensures:
- User can type any number format
- We strip all non-digit characters before parsing
- `parseInt('60000', 10)` always returns `60000` (not affected by locale)

### Step 2: Verify auto-save logic

The auto-save at line 375 already sends `Number(formData.amount)` which is safe since `formData.amount` is now already a proper integer.

### Step 3: Test locally

1. Navigate to a client's subscription tab
2. Enter `60000` in the amount field
3. Wait for auto-save (1 second debounce) or save manually
4. Verify the value `60000` is persisted correctly
5. Test other values like `5000`, `85000`, `100000`
